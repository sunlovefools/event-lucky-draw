# Event Station Quest Lucky Draw Website Spec

## Problem Statement

The event organizer needs a web-based system for delegates to complete vendor station interactions, collect verified station stamps, complete a final event survey, and become eligible for a lucky draw. The current event flow depends on vendors independently confirming station completion, but the organizer needs a reliable way to prevent stamp sharing, monitor progress, manage stations/vendors, and run a public lucky draw from eligible delegates.

Delegates need a low-friction mobile experience: scan the event website QR, identify themselves using their badge QR registration number, enter their full name, see required stations, collect one stamp per active station, complete a final survey, and receive confirmation that they are entered into the lucky draw.

Vendors need a simple portal for their assigned station: generate short-lived one-time stamp QR codes, see when the current QR is consumed, and review who has scanned their station.

Admins need a dashboard to manage the event setup and operations: stations, vendor accounts, admin accounts, delegate progress, station summaries, eligibility overrides, exports, participation open/close controls, and lucky draw control/results.

## Solution

Build a hosted Next.js web application backed by Supabase/Postgres for a single event, while keeping core event configuration database-driven so the system can become reusable later.

The app will support three main authenticated/identified experiences:

- Delegate mobile flow
- Vendor station portal
- Admin dashboard and lucky draw controls

Delegates will identify themselves by scanning or manually entering the registration number from their badge QR. Because the organizer does not currently have access to the delegate registration database, delegates must also enter their full name. The badge registration number becomes the delegate's resume key. After first registration, the browser session should keep the delegate logged in on that device, with badge QR scan/manual registration number as fallback on another device or cleared browser.

Each active station is required. Delegates may complete stations in any order. Disabled stations no longer count as required for lucky draw eligibility.

Vendor station surveys are external/vendor-owned. After the vendor interaction and external survey are complete, the vendor shows a one-time, short-lived QR generated by this system. The QR encodes a secure one-time URL and can be scanned either by the delegate's in-browser scanner or by the phone's native camera app. If the delegate is not registered yet, the app should require badge registration first and then apply the pending stamp.

Stamp QRs are manually generated/regenerated by vendors. They expire after a short duration and are consumed on any scan attempt, including duplicate scans by a delegate who already completed that station. If two delegates scan the same valid QR at nearly the same time, the first successful backend transaction wins; later attempts see that the QR was already used. Expired, used, or invalid QRs show a clear error telling the delegate to ask the station staff for a new QR.

After all active station stamps are collected, delegates complete a fixed built-in final event survey. Survey submission is allowed once only and marks the delegate eligible for the lucky draw. After eligibility, the delegate sees a simple confirmation page and does not see winner results on their phone.

Admins control the public lucky draw from the admin dashboard. The public display is passive, projector-friendly, and polls for current draw state. It runs a simple animation before revealing the winner. Winner history is permanently recorded with flexible draw labels, and previous winners are excluded from future draws by default. Admins can manually include, remove/disqualify, or edit delegate names in the lucky draw pool via delegate records.

## Implementation Decisions

- The project will be built as a hosted web application using Next.js and Supabase/Postgres.
- The first version targets one event, but event setup data such as stations, vendors, active station status, survey questions, delegates, stamps, eligibility, and winner history should live in the database rather than being deeply hardcoded.
- Admins and vendors authenticate with simple username/password credentials. No email, OTP, SSO, or magic-link dependency is required.
- Multiple admin accounts are supported. Complex admin roles are out of scope for MVP.
- Each vendor account is assigned to exactly one station.
- Admins can create/edit stations and vendor accounts from the dashboard.
- Delegates do not create password accounts. Their badge registration number is their resume key.
- Delegate browser sessions should persist after first registration, with badge scanning or manual registration number entry as fallback.
- Delegate full name is manually collected because the badge QR only provides a unique registration number and the external registration database is unavailable.
- The same badge registration number resumes the existing delegate record and progress immediately.
- Active stations define the required completion set. Disabled stations no longer count as required.
- Delegates can complete stations in any order.
- Vendor station surveys are external to the system. The system only verifies station completion through stamp QR collection.
- The final event survey is built into the website, fixed/simple for this event, and submitted once only.
- Submitting the final survey marks a delegate eligible for the lucky draw if station requirements are complete.
- Station stamp QRs encode secure one-time URLs.
- Station stamp QRs must be usable through both in-browser QR scanning and native phone camera scanning.
- If a station stamp URL is opened before delegate registration, the pending stamp should be retained through registration and applied afterward.
- Badge QR scanning must have a manual registration number entry fallback.
- Vendor-generated stamp QRs are short-lived and one-time use.
- Vendors manually generate/regenerate QRs. There is no automatic QR rotation requirement, but a vendor can regenerate before expiry to invalidate the old QR.
- QR consumption is triggered by any scan attempt, including duplicate station scans by a delegate who already has that station stamp.
- Duplicate station scans never award duplicate stamps.
- If two delegates scan the same valid QR concurrently, the backend enforces first-request-wins with database-level transactional safety.
- Expired, used, and invalid QRs show delegates a clear error telling them to ask the station staff for a new QR.
- Vendor QR status updates by polling every few seconds rather than WebSockets/realtime.
- Vendors can see full scan history for their own station, including delegate full name and timestamp.
- Admins can view participant progress by delegate and station summary by station.
- Admins can view full scan audit history including delegate, station, timestamp, QR token reference, scan result, and consumption status.
- Admins can edit delegate full names.
- Lucky draw eligibility is managed through delegate records, not a separate manually maintained draw pool.
- Delegate draw statuses should support at least normal eligible, manually included, removed/disqualified, and winner.
- Admins can manually include or remove delegates from the lucky draw pool.
- Admins can open, close, and reopen participation.
- Closing participation blocks new delegate registrations, stamp collection, final survey submissions, and vendor QR generation.
- Closing participation does not remove already-eligible delegates from the draw pool.
- Admins can export participants/progress, station completions, final survey responses, winner history, and scan audit logs.
- Lucky draw is controlled from the admin dashboard.
- The public lucky draw display is passive and projector-friendly.
- The public lucky draw display polls for current draw state.
- The public lucky draw display runs a simple name-cycling or suspense animation before revealing the winner.
- Winner history is permanently recorded with timestamp and flexible draw label.
- Previous winners are excluded from future draws by default.
- Delegates do not see lucky draw results on their own phone in MVP.
- The app targets medium event scale: 100–1,000 delegates and 10–50 stations.
- Proven third-party QR generation and browser QR/camera scanning libraries may be used.

## Testing Decisions

- Tests should focus on external behavior and business rules rather than implementation details. They should prove that users can complete the required journeys and that the backend enforces the event rules under realistic conditions.
- The main testing seams are end-to-end app behavior and database/service business logic.
- End-to-end tests should cover complete user journeys across delegate, vendor, admin, and public display flows.
- End-to-end delegate tests should verify registration by badge/manual number, name entry, session resume, progress display, valid stamp collection, invalid QR errors, final survey submission, and eligibility confirmation.
- End-to-end vendor tests should verify login, QR generation, QR regeneration, QR status polling, consumed/expired status display, and station scan history.
- End-to-end admin tests should verify login, station/vendor management, participant table, station summary, participation open/close/reopen controls, eligibility overrides, exports, draw control, and winner history.
- End-to-end public display tests should verify passive polling behavior, simple animation state, and winner reveal after admin starts a draw.
- Database/service tests should cover the highest-value business rules directly:
  - active station eligibility calculation
  - disabled station no longer required
  - final survey submit-once behavior
  - one-time QR consumption
  - expired QR rejection
  - used QR rejection
  - duplicate station scan consuming QR but not awarding duplicate stamp
  - first-request-wins QR consumption under concurrent scan attempts
  - pending stamp application after delegate registration
  - participation closure blocking registration/stamping/survey/vendor QR generation
  - lucky draw pool derivation from delegate statuses
  - manual include/remove overrides
  - previous winner exclusion
  - winner history persistence
- Since the repo is currently empty, there is no existing test prior art to follow. The implementation should establish the test structure around the selected Next.js/Supabase stack.

## Out of Scope

- Multi-event admin UI for creating and switching between many events.
- Full reusable event platform behavior beyond database-driven setup for this first event.
- Vendor/station survey builder or vendor survey hosting inside this system.
- Admin-configurable final survey builder.
- Parsing delegate name/company/email from badge QR payloads.
- Integration with an external delegate registration database.
- Delegate password accounts, PINs, OTPs, magic links, or email login.
- Complex admin roles such as super admin, read-only admin, or event staff role permissions.
- SSO or enterprise authentication.
- Dedicated scanner devices or venue-local server deployment.
- Complex lucky draw prize management.
- Undo/redraw workflow for winners.
- Delegate-facing winner/result pages.
- Advanced realtime infrastructure unless later implementation proves polling insufficient.
- Complex lucky draw visuals beyond a simple animation.
- Offline-first behavior for poor venue internet.
- Automatic QR rotation without vendor action.
